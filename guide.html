Features

  • Export to specific APIs document which’s extension is .doc.html and you can view it offline
  • Auto play test via npm, you can integrate into jenkin…
  • Use variable to store response data which will be reuse for the testcases
  • See variable value via console log in browser by click on variable which you want to debug
  • Generate result file to html which so easy to view and debug

How to install

              $ git clone https://github.com/just4developments/autotestapi.git autotestapi
              $ cd autotestapi
              $ npm install

Write testcases

We create a main testcase which will config for a project.
In the maintest case we will declare project name, child testcases...

testcases/main.json

{
  "project": "Project name", // 
  "team": "Dev team",        // Project name which is visibled on footer in the result page
  "env": {},                 // Declare global variables which will be used for all testcases
  "doc": {},                 // 
  "status": [],              // Http response status which you expect for the apis. 
                                It's global, you can overide it for each api by add status field in api in a testcase
  "headers" : {},            // Set global headers in API request, you can overide it for each testcase too
  "testcases": []            // Declare your testcases file 
}
  • project It's Project name which will be visibled on header in the result page or document page
  • team It's your company name or team name which will be visibled on footer in the result page or document page
  • env Global environment variable. Declare global variables which will be used for all testcases
  • doc Declare api description for APIs specification document
  • status Set global http response status which you expect for the apis. You can overide it for each api by add status field in api declaration
  • headers Set global headers in API request, you can overide it for each testcase too
  • testcases Declare your testcases file
"env": {
  "url": "http://localhost",
  "port": 8080,
  "login": {
    "username": "username",
    "password": "123456"
  },
  "someTypes": ["type1", "type2", "type3"]
}

Example: In the testcases, you want request to address: http://localhost:8080.
But in a nice day, you publish your project on some hosting then you want to test on it.
Ofcourse, you will have to change all your testcases from localhost to new address
I think it's not good.
If you set it by environment variable, you only have to change this variable
In url request you change http://localhost:8080 to ${env.url}:${env.port} And to get environment variable in testcase, you can do

  • ${env.url} to get "http://localhost"
  • ${env.port} to get "8080"
  • ${env.login.username} to get "username"
  • ${env.login.password} to get "123456"
  • ${env.someTypes[0]} to get "type1"
  • ${env.someTypes[1]} to get "type2"
